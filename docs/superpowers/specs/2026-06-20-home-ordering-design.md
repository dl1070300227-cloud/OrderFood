# 家用点菜小程序第一版设计

## 背景与目标

本项目第一版先做成网页原型，后续保留迁移到微信小程序的空间。目标是服务家庭吃饭前的点菜决策：家人可以查看菜品、价格和制作教程，把菜加入本次餐单，提交后生成订单记录。

第一版重点完成从“维护菜品”到“点菜下单”再到“查看历史订单”的闭环。暂不做账号登录、支付、图片上传、云同步和微信小程序原生工程。

## 技术架构

项目采用本地全栈网页应用：

- 前端使用 React/Vite 单页应用，提供点菜、菜品管理、订单记录和教程查看界面。
- 后端使用 Node.js API 服务，负责菜品、教程和订单的增删改查。
- 数据库存储使用 SQLite 本地文件，便于长期保存、备份和后续迁移。
- 本机启动服务后电脑浏览器可访问；后续可配置局域网地址，让同一 Wi-Fi 下的手机访问。

API 与数据模型围绕菜品、教程、订单拆分，避免前端和数据库直接耦合。后续迁移微信小程序时，可以复用后端接口与数据结构。

## 核心模块

### 点菜工作台

点菜工作台是默认首页。页面展示菜品列表，支持搜索和按分类筛选。每个菜品显示名称、分类、价格、简介、预计耗时、难度和推荐状态。

用户可以把菜品加入本次餐单，在餐单里调整数量，填写就餐人数和备注。提交餐单后，系统生成订单记录。

### 菜品管理

菜品管理支持新增、编辑和删除菜品。菜品字段包括：

- 菜名
- 分类
- 价格
- 简介
- 预计耗时
- 难度
- 是否推荐

每个菜品绑定一个文字教程。编辑菜品时可以同时编辑教程内容。

### 制作教程

制作教程采用结构化文字内容，包含：

- 食材
- 调料
- 步骤

用户在点菜时可以快速查看教程；在菜品管理中可以维护教程。第一版不做图片和视频，但数据结构保持后续扩展空间。

### 订单记录

订单记录展示历史点餐结果，默认按下单时间倒序排列。订单字段包括：

- 下单时间
- 菜品明细
- 就餐人数
- 备注
- 总价
- 完成状态

订单可以在未完成与已完成之间切换，方便区分刚点完和已经做完的餐。

## 数据流

应用启动时，前端从后端读取菜品列表和最近订单。用户在点菜工作台中选择菜品时，本次餐单先保存在前端状态里。

提交订单时，前端把餐单、就餐人数和备注发送给后端。后端根据当前菜品价格生成订单快照并保存到 SQLite。历史订单保存菜名和价格快照，因此后续菜品改名、改价或删除，都不会影响历史订单。

菜品管理中的新增、编辑和删除操作直接写入 SQLite。教程随菜品读取和保存。

## 数据模型

### `dishes`

保存菜品基础信息：

- `id`
- `name`
- `category`
- `price`
- `description`
- `estimated_minutes`
- `difficulty`
- `is_recommended`
- `created_at`
- `updated_at`

### `recipes`

保存菜品制作教程：

- `id`
- `dish_id`
- `ingredients`
- `seasonings`
- `steps`
- `created_at`
- `updated_at`

### `orders`

保存订单主信息：

- `id`
- `ordered_at`
- `diners_count`
- `note`
- `total_price`
- `status`
- `created_at`
- `updated_at`

`status` 第一版使用 `pending` 和 `completed`。

### `order_items`

保存订单菜品快照：

- `id`
- `order_id`
- `dish_id`
- `dish_name`
- `quantity`
- `unit_price`
- `subtotal`

`dish_id` 用于追溯原菜品；`dish_name`、`unit_price` 和 `subtotal` 用于保护历史订单快照。

## 校验与异常处理

菜品校验规则：

- 菜名不能为空。
- 价格必须是非负数字。
- 预计耗时如果填写，必须是非负整数。
- 教程步骤可以为空；为空时前端显示“暂无教程”。

订单校验规则：

- 餐单为空时不能提交订单，并在页面显示提示。
- 就餐人数默认是 `1`，必须大于等于 `1`。
- 订单总价由后端按快照统一计算，不信任前端提交的总价。

交互规则：

- 删除菜品前显示确认。
- 已有历史订单的菜品仍允许删除，因为订单保存的是快照。
- 后端接口失败时，前端显示简短错误提示，并保留用户当前输入和餐单状态。
- 首次启动时自动写入一批示例家常菜，方便打开后立即试用。

## 测试与验收

后端测试覆盖：

- 菜品新增、编辑、删除和查询。
- 教程创建与更新。
- 订单创建。
- 订单明细快照。
- 订单完成状态切换。

前端验证覆盖：

- 项目构建。
- 基础类型检查。
- 点菜、提交订单、查看订单等核心流程的手动验收。

手动验收路径：

1. 启动前端和后端。
2. 新增一道菜并填写教程。
3. 在点菜工作台查看这道菜和教程。
4. 将菜加入餐单，调整数量。
5. 填写就餐人数和备注。
6. 提交订单。
7. 在订单记录中查看订单。
8. 将订单标记为已完成。

## 非目标

第一版不包含：

- 用户账号和登录。
- 支付流程。
- 图片上传。
- 视频教程。
- 云端同步。
- 微信小程序原生工程。
- 复杂统计报表。

这些能力可以在第一版闭环稳定后逐步追加。
